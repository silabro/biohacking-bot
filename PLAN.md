# Master Architecture Document: "Silabro Biohacking Bot"

Данный проект — это не просто счетчик калорий. Это **data-driven Health Advisory Board (Многоагентный консилиум)** для биохакеров. Проект сфокусирован на интервальном голодании, трекинге нутрицевтиков (БАДов), уровне энергии и научно обоснованной клинической нутрициологии с Agentic Reflection и динамическим планированием.

---

## 1. Архитектура проекта и стек технологий (Backend Resilience)

### Основной стек:
*   **Язык программирования:** Python 3.12+ (строгая типизация).
*   **Фреймворк для Telegram бота:** `aiogram 3.x` (полностью асинхронный).
*   **Управление состоянием (FSM):** Для экономии токенов и жесткого контроля, дорогие сценарии (ввод веса, регистрация профиля) реализованы через машину состояний (`aiogram.fsm`). На продакшене используется исключительно `RedisStorage` (через `redis:alpine`).
*   **Интеграция с AI:** `google-generativeai` (Gemini 1.5 Flash/Pro). Базовый клиент (`gemini_client.py`) обернут библиотекой `tenacity` для обработки rate limits (HTTP 429) с graceful degradation.
*   **База данных:** ИСКЛЮЧИТЕЛЬНО PostgreSQL + `asyncpg` с первого дня разработки (в т.ч. в докере для тестов). В связи с активным использованием типа `JSONB`, использование SQLite **строго запрещено**, так как ее отсутствие сломает ORM-модели.

### Strict Engineering Constraints & Fault Tolerance:
1.  **Global Localization Rule:** Любая коммуникация ИИ, системные промпты, логи и инлайн-кнопки — **строго на русском языке (Strict Russian UI)**.
2.  **Test-Driven Development (TDD):** Покрытие кода тестами **>= 80%** (GitHub Actions).
3.  **The Async Trap (CPU-Bound Protection):** Операции, требующие высоких вычислений процессора (парсинг мега-JSON файлов), **строго запрещено** выполнять в основном потоке. Они обязаны выгружаться в отдельный пул потоков: `await asyncio.to_thread()`, чтобы предотвратить блокировку Event Loop и зависание бота. Генерация PDF отчетов переведена на связку `WeasyPrint` + `Jinja2`, обернутую в **выделенный изолированный ThreadPool** (на 3-4 воркера), чтобы тяжелый рендер не заблокировал дефолтный `ThreadPoolExecutor` асинхронного цикла (Защита от Thread Starvation).
4.  **Database Thread Safety (AsyncpgEngine Pool):** `NullPool` СТРОГО ЗАПРЕЩАЕТСЯ для основного Telegram-бота, так как это приведет к connection exhaustion при высоких нагрузках. Для `AsyncpgEngine` необходимо использовать стандартный пул соединений с лимитами (например, `pool_size=20, max_overflow=10`).
5.  **Data Access (Unit of Work):** Паттерн доступа — **Repository Pattern**. Любые операции, изменяющие несколько таблиц (например, парсинг чека + Moving Average), оборачиваются в строгую транзакцию `async with session.begin():` (полный коммит или полный роллбэк).
6.  **The Long-running Transaction Trap (Pool Exhaustion):** Жестко запрещается удерживать открытую транзакцию БД во время HTTP-вызовов к LLM. Сетевые запросы должны выполняться ВНЕ блока `async with session.begin():`. Транзакция БД открывается исключительно на доли секунды в самом конце пайплайна для быстрого сохранения результатов.
7.  **Dishka DI Scopes (Connection Leak Protection):** `AsyncEngine` инициализируется как `Scope.APP` (живет весь цикл приложения), в то время как `AsyncSession` и все репозитории **ОБЯЗАНО** используют `Scope.REQUEST` (создаются и закрываются на каждый апдейт/сообщение) для предотвращения exhaustion пула соединений.
8.  **LLM Rate Limiting:** Защита бюджета Gemini API от DDoS-спама через Redis-based `ThrottlingMiddleware` в `aiogram` (возвращает оставшееся время ожидания).
9.  **External API Caching (Rate Limit Protection):** Все успешные ответы от внешних API (USDA, Open Food Facts) кэшируются в Redis (`api:cache:{product_name}`) с TTL 7 дней (604800s). Клиент обязан проверять кэш перед HTTP запросом.
10. **Smart Broadcaster (FloodWait Protection):** При рассылке уведомлений (утром/вечером) строго запрещено использовать сырой `asyncio.gather` для всей базы. Рассылка должна происходить батчами (например, по 25 сообщений) с принудительным `asyncio.sleep(1)` между ними для соблюдения лимитов Telegram.
11. **Prompt Template Registry (Секция 3):** Системные промпты агентов ЗАПРЕЩЕНО хардкодить в Python-строках. Они должны храниться в отдельной директории `prompts/` в виде `.jinja2` шаблонов для динамической инъекции контекста.
12. **Structured Outputs (JSON Schema):** Агенты Консилиума не общаются сырым текстом (Markdown). Метод агента `generate_response()` обязан запрашивать у Gemini строго типизированный JSON ответ, который Python парсит в Pydantic-модели (DTO). Итоговое сообщение для Telegram собирает Python-код из этих DTO.
13. **Application-Level Encryption:** Хранить OAuth-токены `wearable_tokens` в БД в открытом виде строго запрещено. Перед записью в PostgreSQL они должны симметрично шифроваться через **`MultiFernet`** (обязательно для поддержки ротации ключей) и расшифровываться только в оперативной памяти в момент HTTP-запроса к API умных часов.

## 2. Telegram UI/UX & Frictionless Tracking

1.  **Chrono-nutrition Matrix:** Жесткие рамки "Завтрак/Обед/Ужин" упразднены. Динамическая матрица `meal_schedule` базируется на окне голодания.
2.  **Weekly Strategy & FSM Profiling:** Меню разделено на создание недельных Протоколов (`[🔬 Создать Протокол]`), ежедневную калибровку (`[🍽 Рацион на сегодня]`) и настройки профиля с FSM-меню `[🥗 Мое питание]` и `[🏋️ Мой спорт]` (сбор инвентаря и травм).
    *   **Calibration Protocol (Вилка Калибровки) & Shadow Mode:** В онбординге для новых пользователей добавлен выбор: `[🔬 Начать 7-дневную калибровку]` (для новичков, сбор базлайнов RHR, VO2 Max) ИЛИ `[⚙️ Я профи (Ввести метрики вручную)]` (быстрый старт без ожидания). Первым `user_protocols` устанавливается выбранный вариант. Во время калибровки бот работает в **Shadow Mode**: ежедневно выдает рационы по безопасной усредненной формуле (Mifflin-St Jeor) и присылает "Микро-задания" для вовлечения, пока собираются данные.
3.  **Clinical Timeline UI & Meal-Anchored Supplements:** Markdown-хронология с точными таймингами. БАДы жестко привязаны к приемам пищи (например: "За 30 мин до еды"). Под каждым приемом пищи выводятся две кнопки: `[✅ Съел строго по плану]` (списывает всё и логирует БАДы) и `[💬 Изменить факт]`. При нажатии на `[💬 Изменить факт]` юзер пишет (например: "Съел всё, но забыл выпить Омегу"), и бот через FSM+NLP точечно корректирует логирование, чтобы не искажать медицинскую статистику.
    *   **Smart Chunking (MessageIsTooLong Protection):** Если итоговый текст ответа Консилиума превышает лимит Telegram в 4000 символов, бот ОБЯЗАН использовать **HTML-разметку** и разбивать текст через парсер (например, `BeautifulSoup`), который гарантирует безопасное автоматическое закрытие тегов (`</div>`, `</b>`) при разрыве сообщения, исключая ошибку `Can't parse entities`.
4.  **Redis Draft State & Hybrid Editing (Graceful Expiration):** Текущий сгенерированный рацион сохраняется в **Redis** (`draft_ration:{user_id}`).
    *   **TTL Rule:** Строгий TTL 24 часа (`ex=86400`) для предотвращения утечек памяти (OOM). 
    *   **Fallback UX:** При клике на просроченный черновик бот отвечает: *"Этот черновик устарел. Давай сгенерируем новый!"*.
    *   Пользователь редактирует один прием пищи (NLP или кнопки); система пересчитывает макросы на остаток дня и кэширует новый драфт *без* обращения к БД.
5.  **Anti-Flickering Dashboard, Dynamic Fasting & Cheat Meals:** Мелкие действия обновляют табло через `edit_message_text` (inplace update). Пересоздание вниз чата ("Липкий" Фронтенд) происходит только если юзер прислал новое сообщение (сверка `last_user_message_id`).
    *   **Data Completeness Score:** На дашборде выводится индекс точности ИИ (например: "Точность 65%. Сдай Базовый минимум, чтобы открыть клинические дозировки"). Работа бота не блокируется без анализов.
    *   **Dynamic Fasting Widget:** Виджет голодания (`🟢 Пищевое окно` / `🔴 Голодание`) отображается ТОЛЬКО если `Diagnostic Agent` прописал протокол голодания. **Глубина голодания:** Виджет динамически показывает стадии физиологического процесса (например: "12ч - Кетоз", "16ч - Старт аутофагии", "24ч - Пик аутофагии"). Иначе скрыт. **Contraindication Engine Override:** Если Python-префильтр Contraindication Engine (Секция 3) заблокировал фастинг (например, гастрит), виджет `🔴 Голодание` программно подменяется на `🟢 Режим покоя ЖКТ`, и кнопки управления фастингом скрываются.
    *   **Smart Cheat Meal & Emergency Coach:** Кнопка `[🍔 Незапланированный прием]` выдает обязательную вилку: `[🍽 В ресторане / В гостях]` (калории считаются, но холодильник `fridge_items` НЕ трогается) или `[🏠 Из своих запасов]` (считаются калории и списываются продукты). Если активен виджет `🔴 Голодание`, кнопка `[🍔 Голоден]` заменяется на `[🆘 Очень голоден / Срываюсь]`. При нажатии бот не дает еду сразу, а выдает психологическую поддержку и совет выпить воды с электролитами. Кнопка завершения голодания доступна только после этого шага.
    *   **Progressive Disclosure UI (Спорт & Анализы):** На дашборде тренировка выводится кратко (например: `[18:00] 🏋️ Силовая - 45 мин`). По клику `[🔍 Открыть план]` бот выдает детальный план и кнопку `[💬 Изменить факт / Обсудить]`. Добавлена кнопка `[🩸 Мои анализы / Медкарта]`, открывающая таймлайн: Ожидает сдачи, Плановый (Квартальный), История. Бот выдает списки анализов текстом (моноширным шрифтом для копирования) + статически зашитые в код (без LLM) EBM-правила подготовки к сдаче (голод, отмена тренировок).
    *   Содержит кнопку `[🎒 Мой холодильник / Инвентарь]`, открывающую интерфейс управления остатками с обязательной пагинацией.
6.  **Frictionless Tracking (NLP Cart, Smart Water, Workouts & Medical OCR):**
    *   **Smart Shopping Session & Atomic Commit:** Фото чеков склеиваются по id альбома. Список покупок хранится в Redis как `active_shopping_list:{user_id}`. При загрузке чека или NLP-вводе парсер делает **Dry Run (Превью)**, сохраняя результат в буфер `draft_receipt:{user_id}`. Бот выдает текст с превью: *"Я нашел картофель (1кг). Добавляем?"* и вилку: `[✅ Да, всё точно]`, `[💬 Ошибка OCR (Правки)]`, `[📸 Дослать чек]`. Только при нажатии `[✅ Да, всё точно]` происходит **Atomic Commit** (Двухфазный коммит): `INSERT/UPDATE` в `fridge_items`, вычитание из `active_shopping_list` и удаление буфера.
    *   **Graceful Overdraft & NLP Refund TTL:** Списание из `fridge_items` ДОЛЖНО использовать функцию `GREATEST` (`UPDATE fridge_items SET quantity = GREATEST(quantity - X, 0)`). При возврате объедков через NLP, код ОБЯЗАН делать `UPDATE` существующей записи, прибавляя граммы, а не `INSERT`, чтобы дата `created_at` осталась старой для отслеживания срока годности (гистамин).
    *   **Smart Water Tracker:** На основе истории `water_preferences` (JSONB) выводится кнопку `[💧 +250 мл]`. При логировании добавлен переключатель `[💧 Обычная вода]` / `[⚡️ Электролиты/Минеральная]` (критично для предотвращения вымывания солей на голодании).
    *   **1-10 Symptom Tracker & Wearables MVP:** Пользователь логгирует симптом ("Болит живот"). ИИ запросит интенсивность (1-10). В онбординге бот предлагает подключить умные часы (Huawei/Terra) через паттерн Value-Exchange: *"Подключи часы, чтобы Консилиум видел твой сон и шаги"*.
    *   **Manual/OCR Workout Logging (Graceful Degradation):** Интент `WORKOUT_LOGGING`. Юзер присылает скриншот Strava/Apple Fitness (парсинг Gemini Vision) ИЛИ текст вручную. При ручном вводе бот запрашивает RPE (1-10) и длительность, высчитывая `sRPE Load` (RPE * минуты) для эмуляции `Training Load`. **Асинхронный Триггер:** Сразу после логгирования новой тренировки (без ожидания крона) математически высчитывается ACWR (Acute:Chronic Workload Ratio).
    *   **Глобальная отмена (Удаление "зомби-черновиков"):** Во ВСЕХ FSM-состояниях ожидания и при работе с Redis-черновиками (чеки, анализы, редактирование меню) ОБЯЗАТЕЛЬНО должна присутствовать инлайн-кнопка `[🗑 Отменить / Сбросить]`, мгновенно удаляющая ключ черновика из Redis и возвращающая в главное меню.
7.  **Гибкая Ревизия Холодильника (Smart Fridge Audit):** При отключении `Travel Mode` (возвращении домой) после отсутствия >4 дней, бот выгружает `fridge_items` в фоновый промпт LLM для предсказания порчи продуктов. Бот выдает список скоропортящихся продуктов с кнопками: `[✅ Списать предложенное]`, `[💬 Внести правки ]` (переход в NLP-парсер) и `[❌ Ничего не испортилось]`.
8.  **Redis UX-Locks & Idempotency (Защита от спама и гонок):** Внедрены строгие замки БД для тяжелых LLM-генераций и защита от двойного списания продуктов. Постановка замка на 60 сек (`SET lock:{action} 1 EX 60 NX`), в блоке `finally` замок программно удаляется (`DEL`). Если юзер кликает во время лока, бот выдает Telegram Toast (всплывашку): *"⏳ Обрабатываю запрос..."*. Перед записью в БД обязательна **Идемпотентность**: `SELECT` проверка статуса (например, "Съедено"), чтобы прервать дублирующую транзакцию, спасая БД.
9.  **PDF Medical Report Export:** Генерация красивого Markdown->PDF отчета врача (проверка на Empty-State, подтверждение/комментарии пациента перед окончательным рендером файла).

## 3. Multi-Agent Orchestration (The "Consilium" Pipeline)

Архитектура принятия решений строится на AI-пайплайне с передачей контекста:
*   **Стриминг статусов (UI-Amortization) & Cancellation Token (Стоп-кран):** Так как 4-5 вызовов LLM занимают время, бот ОБЯЗАН амортизировать ожидание через `edit_message_text` со статусами: *"👨⚕️ Врач анализирует... ➡️ 👨🍳 Повар подбирает рецепт..."*. Обновление статуса СТРОГО не чаще 1 раза в 3 секунды. Во время стриминга под сообщением должна быть кнопка `[🛑 Отменить генерацию]`. При ее нажатии сервер обязан прервать HTTP-сессию с Gemini (через `aiohttp` timeout/cancel task) и досрочно снять `Redis Lock`, возвращая юзера в главное меню.
*   **Context Injection Rule & Prompt Template Registry:** Поле `dietary_preferences` инжектируется в System Prompt ИСКЛЮЧИТЕЛЬНО для агентов Консилиума. Промпты агентов хранятся в `prompts/` как `.jinja2` шаблоны. Системные агенты этот контекст НЕ получают для экономии токенов.
*   **Consilium StateGraph & Blackboard Pattern:** Отказ от передачи сырого текста между агентами. Внедрен паттерн **Blackboard (`ConsiliumState`)**. Агенты (Врач, Тренер) генерируют в `ConsiliumState` строго типизированные JSON-ограничения (Constraints), например `{"banned_ingredients": ["сахар"], "target_calories": 2000}`. Python-код (Оркестратор `ConsiliumOrchestrator`) физически применяет эти фильтры к БД (исключая запрещенку) ДО вызова Шеф-повара. Повар не может сгаллюцинировать, так как работает с уже очищенными данными.
*   **Structured Outputs:** Ответы агентов возвращаются в виде строго типизированного JSON, который парсится в `Pydantic` DTO. Итоговое сообщение собирается на стороне Python.
*   **Флоу передачи контекста Консилиума (Ускорение):** Huawei API / sRPE -> `ConsiliumOrchestrator` -> вызовы `Diagnostic` и `Performance` запускаются **ПАРАЛЛЕЛЬНО**, снижая Latency -> готовые Constraints пишутся в Blackboard -> `Biohacker` (назначает спорт-пит) и `Culinary` (добавляет ккал в рацион) -> `Auditor`.

1.  **Diagnostic Agent (Врач & Эндокринолог):** Формирует базовые медицинские правила. **Metabolic Stress & RED-S Logic:** Рассчитывает `Metabolic Stress Score` (кластерный анализ: падение температуры, нарушения сна, падение HRV). При риске RED-S (или тиреоидной адаптации) Врач ПРИНУДИТЕЛЬНО приостанавливает фастинг и назначает рефид. **Женская физиология (Эндокринолог):** Превентивно корректирует норму железа в менструацию и повышает калораж в лютеиновую фазу. В позднюю лютеиновую фазу (данные из `menstrual_cycle_data`) Врач ПРИНУДИТЕЛЬНО блокирует интервальное голодание свыше 12 часов для защиты прогестерона и блокировки скачка кортизола. **Адаптивный Deload:** При жалобах дает вилку решения. **Защита от Орторексии:** При паттернах навязчивого контроля Врач принудительно предлагает **"Mindful Days"** (без граммовок, только визуальный контроль тарелки).
    *   **Contraindication Engine (Жесткий Python-префильтр):** ПЕРЕД вызовом LLM Врача, Python-код анализирует поле `diseases` в БД. Если обнаружены заболевания ЖКТ (например, `"гастрит"`, `"ГЭРБ"`, `"язва"`), код ПРИНУДИТЕЛЬНО пишет в `ConsiliumState` хардкодированные ограничения: `{"max_fasting_hours": 12, "min_meals": 4, "temperature_control": true}`. LLM Врача НЕ МОЖЕТ переопределить эти Constraints. Правила маппинга `disease -> constraints` хранятся в конфигурации `app/core/clinical/contraindication_rules.json`.
    *   **Proactive Checkup Generator (Модульные Чекапы):** В файле `app/core/clinical/checkup_protocols.json` хранятся "Модульные блоки" анализов (например: `"gastritis_block": ["Витамин B12", "Сывороточное железо"]`, `"stress_block": ["Магний в эритроцитах"]`, `"base_minimum": ["Общий анализ крови", "Витамин D"]`). Врач анализирует профиль юзера (`diseases`, `symptom_logs`) и возвращает в Structured Output массив ярлыков: `["base_minimum", "gastritis_block", "stress_block"]`. Python-код склеивает массивы из JSON-файла, убирает дубликаты (`set()`) и сохраняет финальный список в `biomarker_logs` со статусом "Ожидает сдачи".
2.  **Performance Agent (Тренер-Нейрофизиолог / S&C Coach):** Внедряет авторегуляцию (ACWR). **Tissue-Specific Load Management:** Дифференцирует стресс (`stress_type`). Метаболический требует активного восстановления; Неврологический (ЦНС) + `cognitive_load_hours` — абсолютного сна; Механический — нутритивной поддержки. **Adaptive Rehab Protocol:** Если `pain_category` = Structural/Tendon, Тренер исключает динамические векторы нагрузки, активируя изометрию/эксцентрику (Tendon Neuroplastic Training) и мобильность. При `Travel Mode` меняет инвентарь на Bodyweight.
    *   **The Ramp-Up Protocol & Micro-Habits (Zero-to-One):** Для пользователей с `activity_level` = `low` / `sedentary` Тренер активирует режим **"Zero-to-One"**. Полноценные тренировки БЛОКИРУЮТСЯ. Назначаются только NEAT-активности (Activity Snacks: прогулки, стояние, лестница), которые логируются в `lifestyle_logs` (как `movement_minutes`), а НЕ в `workouts_log`, чтобы защитить метрику ACWR от аномального скачка. **Геймификация прогрессии (Ramp-Up):** Неделя 1-2: только прогулки и NEAT. Неделя 3: Python автоматически добавляет изометрию и суставную гимнастику (подготовка связок). Неделя 4 (Выпускной): Тренер программно повышает `activity_level` юзера в БД, поздравляет его и открывает полноценный модуль тренировок с расчетом ACWR и тоннажа.
3.  **Biohacker Agent (Биохакер):** Назначает БАДы. **Хронофармакология:** Назначает БАДы с учетом циркадных ритмов (`circadian_anchor` в БД). Например, витамин D — строго утром (AM), магний глицинат — строго вечером (PM). Добавлен **Caffeine Washout**: если RHR растет на фоне стимуляторов, инициируется протокол полной отмены кофеина на 3-5 дней для сброса рецепторов. **Circadian & Environmental Sync:** Анализирует `lifestyle_logs` в связке с метриками часов для поиска сложных корреляций (например: дефицит утреннего `light_exposure` коррелирует с плохим `Deep Sleep`). Использует данные **CGM** для персонализации Food Sequencing. **Safe Baseline Protocol:** Без свежих анализов назначает только безопасную базу, терапевтические дозы заблокированы.
4.  **Culinary Agent (Шеф-повар):** 
    *   **Двухрежимный Шеф-повар & Магия Дельты:** Работает в 2 режимах. **Режим "Выживание" (Just-in-Time):** При запросе `[🆘 Хочу перекусить]`, Python жестко фильтрует `fridge_items` по правилам Врача и отдает Повару только доступные остатки. **Режим "Творчество" (Воскресная Стратегия):** При планировании, холодильник скрыт от ИИ; генерируется идеальное меню по Constraints Врача/Тренера. Затем Python математически *вычитает* сгенерированные ингредиенты из наличия в БД, и только эту "Дельту" формирует в `🛒 Список покупок`.
    *   **Умное Избранное, Macro-Fillers & Graceful Fallback:** При выборе `[🌟 Собрать из Избранного]`, если любимый рецепт заблокирован Врачом, LLM-Повар генерирует новый, ОБЯЗАТЕЛЬНО начиная ответ с объяснения: *"Твои Сырники сейчас не подходят из-за [Причина]. Я придумал безопасную альтернативу!"*. Если безопасный рецепт из Избранного не дотягивает до целевых калорий Тренера, Шеф-повар НЕ МЕНЯЕТ оригинальный рецепт, а добавляет к нему "Macro-Filler" (например, + горсть орехов на гарнир).
    *   **Bioavailability & Sequencing Engine:** Сверяется с `food_matrix_json`. Нейтрализует антинутриенты (фитаты/лектины) правильной обработкой (ферментация, замачивание). Обеспечивает синергию усвоения. **Nutrient Pairing & Disease-Aware Filtering:** Колонка `food_matrix_json` в `fridge_items` обогащается EBM-тегами (например, `high_oxalate`, `high_purine`, `goitrogenic`). Файл `app/core/clinical/nutrient_synergy.json` хранит матрицу правил синергии и антагонизма (например: `{"tag": "high_oxalate", "disease": "МКБ", "action": "add_calcium_source", "reason": "Нейтрализация оксалатов"}`). Python-скрипт (Оркестратор) сверяет теги ингредиентов рецепта с `diseases` юзера по этой матрице и передает Повару корректирующий Constraint: *"Добавь источник кальция для нейтрализации оксалатов"*.
    *   **Glycemic Sequencing:** Генерирует строгий хронологический порядок употребления нутриентов (клетчатка -> белки/жиры -> сложные углеводы).
    *   **Satiety Auto-Regulation & Защита Глубокого сна:** При системно низком `satiety_level` сдвигает фильтр на High-Volume / Low-Calorie Index. Этот алгоритм жестко блокируется за 3-4 часа до сна для защиты HRV. Если юзер жмет `[🍔 Голоден]` поздно вечером, Повар выдает **Low-Volume Sleep-Promoting Snack**.
    *   **Магия Виртуальных контейнеров & Кулинарные коэффициенты (Meal Prep Hack):** При готовке LLM определяет `Dish Type`. Python-код применяет зашитый константный множитель (уварки) для быстрого создания Виртуального контейнера (`🍱 Готовое блюдо` в БД).
    *   **Решение "Проблемы Половины Банана" (Двойной вывод):** `Culinary Agent` обязан возвращать ингредиенты в формате Structured Output (JSON) с двумя полями: `weight_g` и `visual_measure` (например, "половина среднего банана" — для удобного отображения в UI).
5.  **Auditor Agent (Главврач / EBM-Инквизитор):** Критический гейткипер (PubMed/EBM стандарты). Блокирует лженауку (гомеопатия, усталость надпочечников).
    *   **Паттерн Транслятора для Лекарств (RuPharma Architecture):** Для защиты от галлюцинаций LLM выступает ТОЛЬКО переводчиком из Торгового Названия ("Эутирокс") в МНН и далее в **Код АТХ** (Анатомо-терапевтическо-химическая классификация). Проверка на совместимость и CYP450 идет через детерминированное ядро — прямым запросом к `substance_interactions` (ATC Conflict Matrix). Взаимодействия проверяются не по отдельным молекулам, а по классам (ATC кодам).
    *   **State Tracking & Адаптивный Fallback:** При отклонении рецепта Аудитор передает Повару системный флаг: *"Прошлый вариант забракован, сгенерируй принципиально ДРУГОЕ блюдо"*, защищая от бесконечного цикла. 
    *   **Защита Варианта Б (Fallbacks):** При двойном отклонении 2 пути: Вариант А (продукты нарушают мед. протокол) -> идеальный меню из новых продуктов + `🛒 Список покупок`. Вариант Б (сложные макросы) -> базовый "моно-рецепт" строго из `fridge_items`. Если Аудитор переходит к Варианту Б, Повар ОБЯЗАН проверить **Nutrient Density** остатков. Если в базе остались только несочетаемые или пустые продукты (масло и сахар) — бот принудительно переключает на Вариант А (Отправка в магазин), чтобы не кормить юзера мусором.
6.  **Fast-Track Consilium & Educative Fallback:** Для ручных команд ("свари яйца") система обходит Diagnostician, напрямую отдавая `user_protocols.medical_constraints` Шеф-повару, экономя бюджет.
    *   **Educative Friction:** Если у пользователя НЕТ активного `user_protocols`, бот НЕ использует базовые настройки молча. Данные извлекаются через FSM с предложением выбора "Просто дай рецепт" или "Создать протокол".
7.  **Advanced Vision & Унификация OCR (Gemini 1.5 Flash):** Интент `RECEIPT_LOGGING` парсит продукты и цены чеков. **Унификация OCR:** Gemini Vision ОБЯЗАН конвертировать "штуки", "бутылки" и "упаковки" из чеков в базовые граммы/миллилитры на основе веса, указанного на фото упаковки. **Guided Estimation:** Оценка КБЖУ по фото тарелки запрещена; бот требует фото меню доставки для 100% OCR макросов.

## 4. Схема базы данных (SQLAlchemy Models)

*Модели данных жестко нормализованы. Обязательны композитные индексы (Composite Indexes) по `(user_id, date)` для всех таблиц логов.*

**Timezone Nightmare Rule:** Жесткое правило: все даты/время в логах PostgreSQL хранятся ИСКЛЮЧИТЕЛЬНО в UTC (`TIMESTAMP WITH TIME ZONE`).

**JSONB Schema Drift Rule:** Запрещено использовать `default=None` при добавлении новых полей в Pydantic-схемы. Разработчик обязан применять **Sensible Defaults** (например, `fiber: float = 0.0`) и паттерн **On-the-fly Migration**: использовать декоратор `@model_validator(mode='before')` в Pydantic для перехвата и обогащения старых записей из БД недостающими ключами "на лету" перед валидацией. Использование GIN-индексов обязательно.

1.  **`users`**: `id` (PK), `telegram_id` (Unique), `has_accepted_policy`, `city`, `timezone` (String, default: "Europe/Moscow"), `is_traveling` (Boolean, default: False), `age`, `gender`, `height`, `weight`, `activity_level`, `calories_goal`, `active_ration_id`, `baseline_sleep_hours`, `dietary_preferences` (JSONB), `diseases`, `allergies`, `kitchen_equipment` (JSONB), `fitness_equipment` (JSONB), `fitness_capacity` (JSONB - `baseline_rhr`, `vo2_max`, `lactate_threshold`, `mrv_sets`, `acwr_rolling`, `predicted_1rm`, `red_flags`), `meal_schedule` (JSONB), `water_preferences` (JSONB), `wearable_tokens` (JSONB - шифруется симметрично через Application-Level Encryption `MultiFernet`), `menstrual_cycle_data` (JSONB), `data_completeness_score` (Integer, 0-100), `last_active_at` (TIMESTAMP WITH TIME ZONE).
    *   **`daily_ai_requests` (Integer, default: 5):** Лимит списывается **ТОЛЬКО** при полном запуске Консилиума. *Системные фоновые задачи (Утренний дайджест, Стратегия, Smart Fridge Audit) выполняются вне пользовательского лимита и НЕ расходуют квоту.*
2.  **`user_protocols`**: `id` (PK), `user_id` (FK), `goal`, `medical_constraints_json` (JSONB), `active_until`.
3.  **`user_recipes` (Smart Chef Blindspot Fix)**: `id` (PK), `user_id` (FK), `name`, `macros_json` (JSONB), `ingredients_json` (JSONB), `instructions` (Text), `is_estimated` (Boolean - для Cheat Meals), `created_at`.
4.  **`fridge_items`**: `id` (PK), `user_id` (FK), `name`, `calories`, `proteins`, `fats`, `carbs`, `fiber` (Float), `food_matrix_json` (JSONB), `is_supplement`, `is_staple` (Boolean - **Smart Pantry** для бакалеи, ИИ использует свободно, не списывается из БД), `quantity`, `unit` (SQLAlchemy Enum: 'g', 'ml'), `price_per_unit` (Float - Moving Avg), `price_updated_at`, `is_expired` (Boolean, hidden flag). База физически отвергнет попытку записи других мер.
5.  **`substance_interactions` (ATC Conflict Matrix)**: `id` (PK), `atc_code_a` (String - Класс АТХ), `atc_code_b` (String - Класс АТХ), `interaction_type` (String - CYP450_antagonist, absorption_blocker), `description` (Text), `severity` (String).
6.  **`supplements_log`**: `id` (PK), `user_id` (FK), `name`, `dosage`, `circadian_anchor` (String), `taken_at`.
7.  **`medications_log`**: `id` (PK), `user_id` (FK), `name`, `dosage`, `pharmacokinetics_json` (JSONB).
8.  **`symptom_logs`**: `id` (PK), `user_id` (FK), `symptom_name`, `intensity` (Integer 1-10), `pain_category` (String), `satiety_level` (Integer 1-10), `timestamp`.
9.  **`lifestyle_logs` (Вместо water_logs)**: `id` (PK), `user_id` (FK), `water_ml`, `electrolytes` (Boolean), `light_exposure` (Integer - lux/spectrum), `thermal_stress` (String - sauna/ice), `cognitive_load_hours` (Float), `movement_minutes` (Integer - NEAT/Activity Snacks для Ramp-Up Protocol, НЕ попадает в `workouts_log`), `timestamp`.
10. **`workouts_log`**: `id` (PK), `user_id` (FK), `date`, `workout_type` (String), `duration_minutes` (Integer), `stress_type` (String), `planned_workout` (JSONB), `actual_workout` (JSONB), `recovery_interventions` (JSONB), `training_effect` (Float), `training_load` (Float), `recovery_time_hours` (Integer).
11. **`biomarker_logs`**: `id` (PK), `user_id` (FK), `biomarker_name` (String), `value` (Float), `unit` (String), `reference_min` (Float), `reference_max` (Float), `date_tested` (Date).
12. **`ru_drugs_registry` (База ЕСКЛП)**: `id` (PK), `trade_name` (String - Торговое название), `inn` (String - МНН), `atc_code` (String - Класс АТХ). Наполняется офлайн-парсером CLI.

    *   *Scope:* **Клининг UI/UX Дашборда:** На главном экране жестко закрепить 3 кулинарных вектора: 1) `[🍽 Мой рацион]` — План на день (выполняется через `[✅ Съел по плану]`). 2) `[📝 Я сорвался / Съел другое]` — Логирование постфактум (Вилка: Ресторан / Из холодильника). 3) `[🆘 Хочу перекусить]` — Just-in-Time генерация из остатков. Если активен виджет `🔴 Голодание`, кнопка 3 краснеет: `[🆘 Срываюсь с голода]` (выдает воду с солью и психологический саппорт).
    *   **Магия Контейнеров & Smart Pantry:** `Dish Type` (SOUP, ROASTED). Python применяет зашитый константный множитель к весу сырых ингредиентов для создания Виртуального контейнера (`🍱 Готовое блюдо`). Колонка `is_staple` используется как **Smart Pantry** для бакалеи (не списывается из БД).
    *   **Smart Cheat Meal & Emergency Coach:** Вилка читмила (Без списания / С полным списанием запасов). Экстренный коуч голодания выдает саппорт перед досрочным прерыванием "пищевого окна".
    *   **NLP Refund TTL & Atomic Commit Shopping:** Списание из `active_shopping_list` и перенос из `draft_receipt` в БД происходит атомарно. Возврат объедков (`refund_ingredients`) обновляет старую запись в `fridge_items` (сохраняя срок годности). Вычет идет через `GREATEST(quantity - X, 0)`.
    *   **Идемпотентность & UX Locks:** Выдача Telegram Toast *"⏳ Обрабатываю запрос..."* при 60-сек замке. `SELECT` проверка статуса Съедено.

### Phase 5: Multi-Agent Orchestration (Health Advisory Board)
*   **Task 5.1: The Consilium Pipeline (Consilium StateGraph, Blackboard Pattern).**
    *   **Consilium StateGraph & Blackboard Pattern:** Отказ от передачи сырого текста между агентами. Внедрен паттерн **Blackboard (`ConsiliumState`)**. Агенты генерируют в `ConsiliumState` строго типизированные JSON-ограничения (Constraints: `banned_ingredients`, `target_calories`). Оркестратор физически фильтрует БД по этим Constraints ДО вызова Шеф-повара.
    *   **Contraindication Engine (Python-префильтр Врача):** ПЕРЕД вызовом LLM Диагноста, Python анализирует `diseases` и пишет хардкодированные Constraints в `ConsiliumState` из `app/core/clinical/contraindication_rules.json`. LLM НЕ МОЖЕТ переопределить эти ограничения (например, `max_fasting_hours: 12` для гастрита). Виджет голодания подменяется на `🟢 Режим покоя ЖКТ`.
    *   **Proactive Checkup Generator:** Врач возвращает массив ярлыков чекап-блоков (`["base_minimum", "gastritis_block"]`). Python склеивает модули из `app/core/clinical/checkup_protocols.json`, дедуплицирует и сохраняет в `biomarker_logs` со статусом "Ожидает сдачи".
    *   **Двухрежимный Шеф-повар & Магия Дельты:** Режим "Выживание" JIT (фильтрация `fridge_items` по Constraints Врача) и Режим "Творчество" (генерация идеального меню со скрытым холодильником, затем вычет "Дельты" недостающих ингредиентов для `🛒 Списка покупок`).
    *   **Bioavailability Engine & Nutrient Pairing:** Оркестратор сверяет теги `food_matrix_json` ингредиентов с `diseases` юзера по матрице `app/core/clinical/nutrient_synergy.json` и передает Повару корректирующие Constraints (например, *"Добавь кальций для нейтрализации оксалатов при МКБ"*).
    *   **Умное Избранное, Macro-Fillers & Pre-filtering:** При планировании предлагать `[🌟 Собрать из Избранного]`. Python проверяет `user_recipes` на соответствие медицинским Constraints Врача. Поддержка **Graceful Fallback** (извинение и выдача альтернативы при блокировке любимого рецепта) и **Macro-Fillers** (добор КБЖУ к Избранному без его изменения).
    *   **The Ramp-Up Protocol (Zero-to-One):** Для `activity_level` = `low`/`sedentary` тренировки блокируются. NEAT логируется в `lifestyle_logs.movement_minutes`. Геймификация: Нед1-2 прогулки → Нед3 изометрия → Нед4 "Выпускной" (повышение `activity_level` и открытие полного модуля ACWR).
    *   **Стриминг статусов & Saga Pattern (Атомарность Консилиума):** Амортизировать ожидание вызовов LLM со статусами. Механизм `[🛑 Отменить генерацию]` обрывает In-Memory Context без сохранения мусора в БД/Redis.
    *   **Решение "Проблемы Половины Банана", Эволюция Glycogen Shift & Smart Timeout Тренера:** `Culinary Agent` возвращает JSON с `weight_g` и `visual_measure`. Автосдвиг тренировки через `APScheduler` (при 2-часовом таймауте ответа юзера). Защита Глубокого сна блокирует объем порций вечером.
*   **Task 5.2: The Auditor (RuPharma, ATC Conflict Matrix & Защита Варианта Б).**
    *   **Escape FSM (Persistent Keyboard):** К КАЖДОМУ ответу бота в дискуссии крепится инлайн-клавиатура `[✅ Утвердить это меню]` и `[❌ Отменить всё]`. Лимит токенов: макс. 5 сообщений.
    *   **RuPharma Architecture (ATC Conflict Matrix):** LLM выступает ТОЛЬКО переводчиком Торговое название -> МНН -> Код АТХ. Взаимодействия препаратов проверяются через детерминированный запрос к `substance_interactions` по **Кодам АТХ**, исключая галлюцинации.
    *   **Защита Варианта Б:** Если Аудитор переходит к базовому рецепту из остатков холодильника, Повар обязан проверить **Nutrient Density** остатков. Если это не сочетаемые ингредиенты (сахар, масло), принудительно переводить в Вариант А (Посылать в магазин).
    *   **State Tracking Аудитора:** Флаг *"Прошлый вариант забракован, создай принципиально другое"*.
*   **Task 5.3: Redis Draft State (OOM Protection) & Hybrid UI.**
*   **Task 5.4: Synergy Checker (Мгновенное уведомление).**
*   **Task 5.5: Event-Driven Schedulers (Master Cron) & Амнезия ИИ.**
    *   **Отказоустойчивый Cron (Redis JobStore), Anomaly Detection & Защита от спящих (Dormancy Check):** Управление таймзонами через `APScheduler` с `RedisJobStore`. **Dormancy Check:** Если юзер не взаимодействовал с ботом более 3 дней (`last_active_at`), фоновые LLM-задачи замораживаются до его активности. При отправке дайджеста в 08:00, Cron запрашивает умные часы. Если юзер активен уже несколько часов (смещение offset), бот проактивно спрашивает: *"Кажется, ты сменил часовой пояс. Обновить настройки?"*. Это избавляет PostgreSQL от ежеминутного поллинга и гарантирует выживание расписания (Server Restart Resilience).
    *   **Интерактивная Гистаминная Безопасность:** TTL блюд. Проактивно повар ставит в план. Реактивно пуш от Аудитора с "Histamine Watch" (24ч).
    *   **Амнезия ИИ:** Защита от Context Exhaustion. Старые логи `symptom_logs` сбрасываются в `diseases` и не передаются в Консилиум.
*   **Task 5.6: Clinical PDF Generator (Статические шаблоны & WeasyPrint).**
    *   *Context:* Отказ от Playwright для снижения веса и сложности Docker.
    *   *Scope & Constraints:* ИИ (Врач) генерирует ТОЛЬКО текст Executive Summary. Python-код собирает цифры из БД, подставляет их вместе с текстом в статический `HTML/CSS` шаблон через библиотеку `Jinja2`. Готовый HTML рендерится в PDF через библиотеку `WeasyPrint` внутри `await asyncio.to_thread()`. **Защита от Thread Starvation (Изолированный ThreadPool):** Для `WeasyPrint` должен быть создан ВЫДЕЛЕННЫЙ изолированный пул потоков на 3 воркера, чтобы рендер не заблокировал выполнение других фоновых задач. Генерация СТРОГО в io.BytesIO без сохранения на диск.

## 6. QA & DevOps Requirements

*   **TDD Mandatory:** `pytest-cov --cov-fail-under=80`. Юнит-тесты пишутся совместно со слоем DAO/Agents. 
*   Моккирование AI-агентов обязательно во время тестирования State Machines.
*   **Prompt Testing (Golden Dataset & LLM-as-a-Judge):** Обычного TDD недостаточно. В CI/CD пайплайн в папку `tests/evals/golden_dataset/` складываются статичные JSON-файлы сложных клинических кейсов (например, "у юзера гастрит, запретить фастинг").
*   При изменении системных `.jinja2` промптов, CI/CD прогоняет эти сценарии через агентов и использует паттерн **LLM-as-a-Judge** (вторую нейросеть-судью) для проверки соблюдения медицинских ограничений. Деплой блокируется при провале safety-тестов.
